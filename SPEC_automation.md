# 分析師評估自動化 SOP 計劃書 v1

> **一句話目標**：輸入一個頻道 `@handle`，系統自動跑完「抓料 → 判型 → 抽取 → 結算 → 卡片」，產出一張卡 + 一份可審計的決策紀錄。
>
> **公正原則**：自動化 = 公正。所有人走完全相同的程式路徑，任何「針對個人的手動調整」一律禁止；少數需要判斷的地方用同一套規則、且全部留痕可覆核。
>
> 本文承接 `SPEC_unified.md`（三層框架、八欄表、analysts.py 註冊表）。

---

## 0. 設計前提：所有分析師只有兩種「形狀」

做完 4 人後本質浮現——評估路徑只有兩條，router 決定走哪條：

| 型 | 他說什麼 | 抽取 | 結算 | 既有案例 |
|----|----------|------|------|----------|
| **喊單型** (`call`) | 點名個股/產業，多無期限 | 情緒評分（看多事件） | calendar-time 多天期超額；八欄表取最神/最雷各 5 | 10 KOL、股癌 |
| **預言型** (`forecast`) | 賭資產方向，多帶時間窗 | LLM 抽預言（資產+方向+窗） | 方向命中率；八欄表全列、pending | DAC、吳昌華 |

→ 自動化主體 = **判型 + 走對應固定管線**。

## 1. 主流程：`add_analyst.py "<@handle 或頻道URL>" [--name 中文名]`

```
解析頻道(yt-dlp) → 判型(LLM,留痕) → 抓逐字稿(抗限流) → 字幕可得性檢查
  → 路由抽取(情緒 / 預言) → 標的對映+補價格 → 寫註冊表 → 跑引擎+卡片+回歸 → 審計卡
```

| 步驟 | 動作 | 既有可重用 | 狀態 |
|------|------|-----------|------|
| 1 解析頻道 | handle → channel_id、標題抽樣 | `tools_resolve_channels.py` 的 yt-dlp 模式 | ⚠️改 |
| 2 判型 | LLM 看標題+樣本 → `call`/`forecast` + 理由 | 新 `classify_analyst.py` | ❌ |
| 3 抓逐字稿 | 抗限流、增量、續抓 | `fetch_wu_transcripts.py`（已有抗限流） | ⚠️泛化 |

## 2. 三個「判斷閘門」——全自動，但留痕可審

設計原則：**全部自動決定、不暫停等你**；公正靠「規則統一 + 寫進審計卡 + 設定公開可改」三件事保證（先自動、後可審、隨時可改），而非靠人工批准。下面三處是僅有的「需要判斷」之處，但都由規則/LLM 自動完成並留痕：

### 閘門A — 判型（`call` vs `forecast`）
- 方法：LLM 看 ~15 個標題 + 1 份樣本逐字稿 → 自動輸出型別 + 信心 + 理由，**不暫停**。
- 留痕：審計卡記錄理由與信心；信心 < 門檻只是「旗標」供事後覆核，不阻斷流程。
- 公正：錯判可事後改型別重跑，因有留痕故可逆。

### 閘門B — 資料是否足夠
- 中文字幕率 < 50%、或有效影片 < 20 部 → **標「資料不足，未納入評比」**，不硬算。
- 公正理由：不給資料少的人一個虛假分數，也不偷用爛資料。

### 閘門C — 標的解析（三層規則，消滅 proxy 爭議）
標的怎麼變成可算報酬的東西，依「他講的顆粒度」用同一套規則，**大幅降低人工決定**：

| 他講的 | 解析方式 | 要決定嗎 |
|--------|----------|----------|
| **個股**（長榮、台積電） | 直接用該股（全台股 1921 支已快取） | ❌ 零 |
| **產業/主題**（航運、半導體） | **該產業成分股等權籃**（航運＝長榮+陽明+萬海+裕民等權），全在快取 | ❌ 零（LLM 自動生成籃子） |
| **整體市場/單一資產**（美股、比特幣、黃金） | 該市場代表本身（SPY/BTC/GLD）——這是「就是它」非 proxy | ❌ 零 |

- **關鍵改進**：產業主題**不用 ETF proxy**，改用「成分股等權籃」→ 消滅「半導體該用 SOXX 還 SMH」這種 proxy 選擇爭議；籃子成分公開可審。
- **全自動、不暫停**：新產業主題第一次出現 → LLM **自動生成成分清單** → 直接寫進 `sector_basket.json` 並記入審計卡 → 繼續跑。**不卡你確認**。
- 公正怎麼維持：① 成分籃**公開**於 `sector_basket.json`（人人可看）；② 自動生成的決策**記入審計卡**（事後可覆核）；③ 表**可改**——你或任何人不認同就改一行重跑。即「先自動、後可審、隨時可改」，而非「停下來等批准」。
- 個股與整體市場：本來就零決定（名稱→代碼由情緒抽取自動做；市場代表是固定常數）。
- 公正理由：規則統一、籃子公開，沒有任何人用「特調的 proxy」。

## 3. 公正的內建化：把「手寫判斷」換成「規則」

| 現在手動（不公正風險） | 改規則化（公正） | 落地 |
|----|----|----|
| 判決句我手寫（「唯一撐過FDR」「慘輸」） | 規則生成：依 FDR/超額正負/market_bet 查表 | 新 `verdict_rules.py` |
| benchmark 每人臨時定 | 規則：個股比大盤、賭大盤看絕對 | 已有 `is_market_bet` |
| FDR 家族手動分 | 規則：同型同家族自動分 | 引擎已部分支援，泛化成「依型分家族」 |
| proxy 每人臨時選（SOXX? SMH?） | 產業＝**成分股等權籃**，市場＝固定代表 | 閘門C / `sector_basket.json` |
| 代表天期口頭約定 | 常數 `REP_HOLD=20` 寫死全站 | 已有 |

**核心：沒有任何一格是為某個人特調的 → 公正內建。**

### verdict_rules 範例（規則表，套所有人）
```
若 is_market_bet:
    命中率<45% → "方向預測準度低於擲銅板"
    45–55%     → "方向準度與擲銅板無異"
    >55%(且樣本足) → "方向預測有可驗證優勢"
否則(個股/產業):
    FDR存活 且 超額>0 → "嚴謹校正後仍顯著優於大盤"
    原始顯著但FDR淘汰 → "原始顯著，但多重比較校正後消失"
    其餘            → "與大盤無統計顯著差異"
```

## 4. 審計卡（每人一張，公正可被檢查）

`audit/<name>.json`：
```json
{
  "handle": "@wuchangye", "channel_id": "...", "name": "吳昌華",
  "type": "forecast", "type_confidence": "high",
  "type_reason": "標題均為奇門遁甲預測多資產方向，無個股買賣建議",
  "n_videos_scanned": 268, "n_market_related": 166,
  "n_with_subtitle": 116, "subtitle_rate": 0.70,
  "extraction": {"n_predictions": 356, "sample_accuracy": null},
  "tickers_used": {"us":"SPY","crypto":"BTC-USD","gold":"GLD","hk":"EWH","china":"FXI"},
  "benchmark": "BASKET", "fdr_family": "forecast",
  "verdict": "（規則生成）", "included": true,
  "run_at": "2026-06-13"
}
```
→ 任何人都能檢查「他為何被這樣評」，公正可稽核。

## 4.5 資料源規則（拍板：台股 FinMind、國外 Yahoo）

把「名稱→代碼」與「代碼→價格」分開，各用對的源：

| 用途 | 台股 | 國外/幣/金/指數 |
|------|------|----------------|
| **名稱→代碼**（清冊） | **FinMind `TaiwanStockInfo`**（全上市櫃含中文名，取代舊 729 支子集） | 資產層級對映規則（美股→SPY…）；指名美股個股才用別名表 |
| **代碼→價格**（OHLC） | **FinMind**（**保留下市股**，還權價） | **Yahoo (yfinance)** |

**為何台股改 FinMind**：同時修掉兩個公正漏洞——
- 漏洞一：用全市場清冊當解析器 → 新 KOL 喊的冷門股不再「查無此股」。
- 漏洞二：FinMind **保留下市股歷史**（Yahoo 會砍）→ 最爛的喊單（地雷、掏空、下市）不再蒸發 → 不再系統性美化喊單型。

**代價/前置**：需申請 FinMind token（免費，現 `.env` 為空）；重建一次台股價格快取（含下市股）。國外資產維持 Yahoo（FinMind 不涵蓋）。

**還權口徑**：FinMind 與 Yahoo 還權算法可能微異 → 同一市場只用單一源（台股全 FinMind、國外全 Yahoo），不跨源混算同一標的。

## 6. 實作順序（小步、可驗證）

1. **`verdict_rules.py`**：把現有 4 人判決句改成規則生成，確認語意不走樣。（最小、立即提升公正）
2. **台股資料源換 FinMind**：申請 token → 建 `tw_name_map.json`（全清冊）→ 重建台股價格快取（含下市股）。⚠️ 修兩漏洞的根本，且須重跑回歸確認 KOL/股癌數字位移在可解釋範圍（下市股新增會改變部分結果，要記錄為「修正後基準」）。
3. **標的解析規則 + `sector_basket.json`**：個股走規則、市場走固定代表、產業建成分籃。
4. **`classify_analyst.py`**：判型，先用現有 4 人回測（須各自判對）。
5. **`fetch_transcripts.py` / `extract_predictions.py`**：泛化 wu 版吃 channel_id + type。
6. **`ensure_prices.py`**：缺價自動補（台股 FinMind / 國外 yfinance）。
7. **`add_analyst.py`**：串起以上 + 引擎 + 卡片 + 審計卡。
8. **鎖定基準 + 回歸測試**：以 FinMind 重建後的數字為新基準（`baseline_lock.json`），之後任何改動一鍵比對。

## 8. 誠實的限制（寫進方法區）

1. **判型靠 LLM**：可能錯，故留痕可覆核，非黑箱。
2. **抽取準度 88–95%**：公正要求是「同一 prompt、同一門檻對所有人」，求一致非求完美。
3. **單一市況**：回測期多頭，結論不外推空頭。
4. **產業籃成分選擇**：成分股清單影響結果（取幾大、等權與否）；公開於 `sector_basket.json`，可議可改，但比「單一 ETF proxy」更透明、更難黑箱。
5. **資料源還權差異**：台股 FinMind、國外 Yahoo，兩源還權算法或有微異；故同一標的不跨源混算，台股終值與國外終值不直接跨市場比較（本來各卡回測期/市場就不同）。
6. **survivorship 已處理**：改 FinMind 後台股含下市股，喊單型不再被倖存者偏差美化（這是改源的主要動機）。
